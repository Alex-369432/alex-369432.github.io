**Решения по каннибализации:**
- Главная (`/`) и `/shchenki/` не спорят за один и тот же запрос: главная закрывает широкий бренд/трастовый интент ("питомник сиба ину", "купить щенка в питомнике" — рассказ, почему мы), `/shchenki/` — узкий транзакционный интент "что есть в наличии прямо сейчас" (список конкретных щенков с ценами и статусами "забронирован/свободен"). На главной — блок-анонс из 3 щенков со ссылкой "смотреть всех" на `/shchenki/`, без дублирования полного списка.
- `/tseny/` не пересекается с `/shchenki/`: цены даёт по классам и что входит в стоимость (методология), `/shchenki/` — цену конкретного щенка.
- Информационные подстраницы `/o-porode/*` не конкурируют друг с другом — каждая закрывает свой явно отдельный вопрос пользователя.
- Блог не дублирует `/o-porode/*`: справочные "вечнозелёные" статьи о породе живут в `/o-porode/`, а `/blog/` — про процесс (после покупки, дневник выращивания, новости питомника, разовые темы).

## 3. Внутренняя перелинковка

- Хедер: Главная, Щенки, Цены, О породе, О питомнике, Отзывы, Блог, Контакты + телефон/CTA-кнопка "Написать" всегда видны.
- Хлебные крошки на всех страницах, кроме главной (с BreadcrumbList).
- Главная → блоки-анонсы с прямыми ссылками на `/shchenki/`, `/tseny/`, `/o-pitomnike/`, `/otzyvy/`, `/o-porode/`.
- Каждая карточка щенка на `/shchenki/` → ссылка на `/tseny/` (расшифровка класса) и `/kak-kupit-shchenka/`.
- `/o-porode/*` подстраницы перелинкованы кольцом (related-блок "читайте также" 2-3 статьи) + все ведут к CTA "Смотреть щенков в наличии" → `/shchenki/`.
- `/o-porode/komu-podhodit/` и `/o-porode/harakter-plyusy-minusy/` ссылаются на `/kak-kupit-shchenka/` (осознанный выбор → процесс).
- `/roditeli/` двусторонне связана с `/o-pitomnike/` и с карточками щенков на `/shchenki/` (по каждому щенку — ссылка на его родителей).
- Блог → всегда ссылается на релевантный раздел `/o-porode/` (не дублируя тему) и на `/shchenki/`.
- `/tseny/` и `/kak-kupit-shchenka/` ссылаются на `/pitomnik-ili-s-ruk/` из предупреждающих врезок про низкую цену и риски покупки без документов; `/pitomnik-ili-s-ruk/` ссылается обратно на `/tseny/`, `/kak-kupit-shchenka/`, `/shchenki/`, `/roditeli/`. Страница добавлена по итогам самопроверки структуры — см. SEO_AUDIT.md, раздел «Самокоррекция структуры».
- Футер на всех страницах: полный список разделов + юр. страница + соцсети + контакты (NAP — Name/Address/Phone одинаково на каждой странице, важно для Яндекса).

## 5. Контент-принципы (Этап 4)

- Каждая страница пишется на реальный интент, без "воды" и без дублирования одинаковых абзацев между страницами (у каждой — свой ракурс, даже если тема пересекается, напр. характер породы упоминается и на главной, и в `/o-porode/harakter-plyusy-minusy/`, но на главной — 2 предложения-выжимка со ссылкой "подробнее", а не копия статьи).
- Честный тон: отдельно прописываем `/o-porode/komu-podhodit/` с прямым указанием, кому порода НЕ подойдёт — это и трастовый, и SEO-сигнал полноты (E-E-A-T), и снижает процент "щенок не подошёл" уже после продажи.
- FAQ на каждой важной коммерческой странице — короткий (3-5 вопросов), полный список — на `/faq/`.
- Единицы: цены — в рублях, с диапазоном и пометкой "уточняйте актуальную цену", даты помётов — без выдуманных конкретных дат, только форматы "уточняйте у питомника" или реальные, если появятся от заказчика.

## 6. Коммерческие факторы (Этап 5)

Реализуем на сайте:
- Понятная структура цены по классам (`/tseny/`) вместо "цена по запросу" — прямое конкурентное преимущество (см. SEO_RESEARCH §5.1).
- Явные условия: документы (ветпаспорт, метрика/родословная), вакцинация, что входит в пакет.
- Прозрачный процесс покупки по шагам (`/kak-kupit-shchenka/`).
- Реальные фото щенков/родителей (заглушки на этапе разработки — оптимизированы под WebP/responsive, готовы к замене на реальные).
- FAQ, отзывы, контакты, карта проезда (плейсхолдер), мессенджеры.
- Множественные точки CTA ("Написать в WhatsApp/Telegram", "Позвонить", форма обратной связи) — не агрессивные, не более 1 sticky-CTA на мобильном.
- Раздел юридической информации (реквизиты — плейсхолдер, политика конфиденциальности — обязательна из-за форм сбора данных).

## 7. Технический план (Этапы 6-7)

- Чистый semantic HTML5, без SPA/тяжёлого фреймворка — весь контент в исходном HTML (доступен без выполнения JS).
- Один H1 на страницу, логичная иерархия H2-H3.
- `<title>`/`<meta description>` уникальны на каждой странице (см. §4).
- `rel="canonical"` на каждой странице (self-canonical, без параметров).
- Open Graph + Twitter Card на всех страницах (og:title/description/image/url/type).
- JSON-LD: `Organization`/`LocalBusiness` на всех страницах (сквозной), `BreadcrumbList` на всех, кроме главной, `FAQPage` на `/faq/` и там, где реально есть видимый FAQ-блок, `Product`/`Offer`-подобные данные на карточках щенков (через `ItemList`), `Article` на статьях блога.
- robots.txt — открыт для индексации, ссылка на sitemap.xml, закрыты только служебные (напр. `/thank-you/` если появится).
- sitemap.xml — статический, все канонические страницы, без дублей.
- 404.html — понятная страница с навигацией назад.
- Единый формат URL: везде trailing slash, никаких `?id=` и регистронезависимых дублей.
- Изображения: `<img>` с `width/height` (без сдвигов layout → CLS), `loading="lazy"` для всех, кроме LCP-изображения первого экрана, `alt` с реальным описанием (не keyword stuffing).
- CSS/JS — минимальные, без тяжёлых фреймворков и внешних шрифтов с блокирующей загрузкой; системные шрифты или 1 self-hosted шрифт с `font-display: swap`.
- Аналитика: пустые, явно помеченные слоты под Яндекс.Метрику и Google Analytics ID в одном конфиг-файле — без выдуманных ID.

## 8. Yandex-специфика (Этап 7)

- Полные и одинаковые контактные данные (NAP) в футере каждой страницы + Schema `LocalBusiness` с `addressRegion: Москва`/МО.
- Явная региональность в title/H1 (Москва и МО) без переспама.
- Простой, быстрый, не перегруженный виджетами сайт (важно для поведенческих факторов).
- Мобильная версия — приоритет (Mobile First вёрстка).
- Отсутствие переоптимизации: ключи вписаны естественно, разные страницы — разные формулировки, не механическое повторение фразы.
- Готовность к подключению Яндекс.Вебмастера/Метрики — через конфигурационные плейсхолдеры, без автоматической установки счётчиков с фиктивными ID.
